Decision #2 — What Data Flows (Backend → Frontend)

This is the backend change that makes Decision #1 possible. Today the backend discards tool results and metadata. The proposed change sends them through.

Today — data discarded

SSE stream chunks:

delta: { tool_calls: [{ name: "list_issues" }] }
tool_result → suppressed
delta: { content: "There are 12..." }
ResultMessage → logged only
[DONE]

Proposed — data forwarded

SSE stream chunks:

delta: { tool_calls: [{ name: "list_issues" , args: {...} }] }
delta: { tool_result: { id: "...", content: "Found 12...", is_error: false } }
delta: { content: "There are 12..." }
meta: { turns: 3, cost: 0.0012, duration_ms: 2400 }
[DONE]
Why this matters to you: Without decision #2, the frontend has nothing to show. The green lines are the new data that powers the expanded tool details, timing, error messages, and the "3 turns | 2.4s | $0.0012" bar at the bottom.

Decision #4 — Auto-Collapse (5+ Completed Steps)

When an agent runs many tools (common with /npm-review or code analysis), showing every step clutters the view. After 5 completed steps, they collapse into a summary row.

Without auto-collapse (12 steps)

▶ ✓ npm > fetch_package 0.3s
▶ ✓ npm > check_license 0.2s

▶ ✓ npm > check_maintenance 0.3s

▶ ✓ snyk > advisor_score 0.8s

With auto-collapse

✓ 10 10 completed steps 3.8s total
▶ ✓ npm > fetch_package 0.3s
... 9 more steps ...
▶

Generating final report...

▲ Click the summary row to expand

Rule: Running steps (spinner) and error steps (✗) are never auto-collapsed — only completed steps collapse. This ensures you always see what's actively happening and what went wrong.
